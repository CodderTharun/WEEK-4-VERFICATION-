# Git & GitHub

## Introduction to Git

Git is a Distributed Version Control System (DVCS) used to track changes in files and collaborate with multiple developers.

Git helps developers:

- Track code history
- Restore previous versions
- Work on multiple features simultaneously
- Collaborate efficiently
- Manage software development professionally

![Distributed Git Architecture](/IT/git/distributed-git-architecture.png)

---

## Version Control System

A Version Control System (VCS) records changes made to files over time.

### A. Without Version Control

- Previous code versions are lost
- Collaboration becomes difficult
- Bugs are hard to trace
- Recovery becomes impossible

### B. With Git

- Every change is tracked
- Developers can collaborate safely
- Previous versions can be restored anytime

---

## Distributed Version Control System

Git is distributed because every developer has a complete copy of the repository.

### Advantages

- Faster operations
- Offline work possible
- Better backup system
- Easy branching and merging

![Centralized vs Distributed VCS](/IT/git/centralized-vs-distributed-vcs.png)

## File States in Git

Files in Git can exist in three states.

### A. Modified

The file has been changed but not staged.

### B. Staged

The file is prepared for the next commit.

### C. Committed

The file snapshot is permanently stored in Git history.

## Git Reset

Git reset moves HEAD to another commit.

### A. Soft Reset

<Tabs>
  <TabItem label="Bash">
    <Code
      code={`git reset --soft commitID`}
      lang="bash"
      title="Soft Reset"
    />
  </TabItem>
</Tabs>

Keeps:

- Staging area
- Working files

### B. Mixed Reset

<Tabs>
  <TabItem label="Bash">
    <Code
      code={`git reset --mixed commitID`}
      lang="bash"
      title="Mixed Reset"
    />
  </TabItem>
</Tabs>

Keeps:

- Working files

Removes:

- Staging area

### C. Hard Reset

<Tabs>
  <TabItem label="Bash">
    <Code
      code={`git reset --hard commitID`}
      lang="bash"
      title="Hard Reset"
    />
  </TabItem>
</Tabs>

Deletes:

- Staging area
- Working directory changes

### D. Warning

This operation can permanently delete work.

## Best Practices

### A. Commit Frequently

Small commits are easier to manage.

### B. Write Clear Commit Messages

Bad:

<Tabs>
  <TabItem label="Text">
    <Code
      code={`fixed stuff`}
      lang="text"
      title="Bad Commit Message"
    />
  </TabItem>
</Tabs>

Good:

<Tabs>
  <TabItem label="Text">
    <Code
      code={`Fixed authentication bug in login module`}
      lang="text"
      title="Good Commit Message"
    />
  </TabItem>
</Tabs>

### C. Use Feature Branches

Avoid direct work on the main branch.

### D. Pull Before Push

Reduces merge conflicts.

### E. Never Push Secrets

Do not push:

- API keys
- Passwords
- Environment files
