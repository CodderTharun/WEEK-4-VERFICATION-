# Git & GitHub

## Introduction to Git

Git is a Distributed Version Control System (DVCS) used to track changes in files and collaborate with multiple developers.

Git helps developers:

- Track code history
- Restore previous versions
- Work on multiple features simultaneously
- Collaborate efficiently
- Manage software development professionally

![Distributed Git Architecture](/IT/git/distributed-git-architecture.png)

---

## Version Control System

A Version Control System (VCS) records changes made to files over time.

### A. Without Version Control

- Previous code versions are lost
- Collaboration becomes difficult
- Bugs are hard to trace
- Recovery becomes impossible

### B. With Git

- Every change is tracked
- Developers can collaborate safely
- Previous versions can be restored anytime

---

## Distributed Version Control System

Git is distributed because every developer has a complete copy of the repository.

### A. Advantages

- Faster operations
- Offline work possible
- Better backup system
- Easy branching and merging

![Centralized vs Distributed VCS](/IT/git/centralized-vs-distributed-vcs.png)

## Git Workflow

Git workflow consists of three main stages:

```text
Working Directory → Staging Area → Repository
```

### A. Workflow Steps

1. Create or modify files
2. Add files to staging area
3. Commit changes permanently

![Git Workflow Diagram](/IT/git/git-workflow-diagram.png)

---

## File States in Git

Files in Git can exist in three states.

### A. Modified

The file has been changed but not staged.

### B. Staged

The file is prepared for the next commit.

### C. Committed

The file snapshot is permanently stored in Git history.

![Git File States](/IT/git/git-file-states.png)

## The Staging Area

The staging area acts as a temporary preparation zone before committing.

### A. Important Concept

```text
git add creates a snapshot of the current file state
```

Even after staging, files can still be modified.

---

## Undoing Changes

### A. Unstage a File

```bash
git reset HEAD filename
```

Removes file from staging area while keeping modifications.

### B. Discard Changes

```bash
git checkout -- filename
```

### C. Warning

This permanently removes changes.

---

## Removing Files

### A. Remove File Completely

```bash
git rm filename
```

Removes:

- File from working directory
- File from Git tracking

### B. Remove Only from Git

```bash
git rm --cached filename
```

The file remains on the computer but Git stops tracking it.

### A. Compare Working Directory and Staging Area

### B. Compare Staging Area and Last Commit

```bash
git diff --cached
```

### C. Compare Working Directory and Last Commit


## .gitignore

The `.gitignore` file tells Git which files or folders should not be tracked.

### A. Example

```gitignore
node_modules/
.env
*.log
*.pyc
```

### B. Commonly Ignored Files

- Temporary files
- Build files
- Environment variables
- Logs
- Dependencies

Git stash temporarily stores unfinished changes.

### A. Save Changes

```bash

```

### B. Restore Changes


## HEAD Pointer

HEAD is a special pointer that refers to the current branch or commit.

### A. Example

```text
HEAD → main
```

When switching branches, HEAD moves automatically.

---

## Merging

Merging combines two branches together.

### A. Example

```bash
git checkout main
git merge feature-login
```

Git creates a merge commit that combines histories.

![Git Merge Diagram](/IT/git/git-merge-diagram.png)

## Merge Conflicts

Merge conflicts happen when two branches modify the same section of code differently.

### A. Conflict Markers

```text
<<<<<<< HEAD
Current branch code
=======
Incoming branch code
>>>>>>> branch-name
```

### B. Steps to Resolve Conflicts

1. Open conflicting file
2. Edit manually
3. Remove conflict markers
4. Add file
5. Commit changes

```bash
git add filename
git commit
```

---

## Rebasing

Rebasing moves commits from one branch onto another base branch.

### A. Command

```bash
git rebase main
```

### B. Advantages

- Cleaner history
- Linear commit structure

### C. Important Warning

Never rebase shared or public branches.

### D. Reason

Rebasing rewrites commit history and can break collaboration.

![Merge vs Rebase](/IT/git/merge-vs-rebase.png)

---

## Git Reset

Git reset moves HEAD to another commit.

### A. Soft Reset

```bash
git reset --soft commitID
```

Keeps:

- Staging area
- Working files

### B. Mixed Reset

```bash
git reset --mixed commitID
```

Keeps:

- Working files

Removes:

- Staging area

### C. Hard Reset

```bash
git reset --hard commitID
```

Deletes:

- Staging area
- Working directory changes

### D. Warning

This operation can permanently delete work.

## Push, Pull, and Fetch

### A. Push

Uploads commits to remote repository.

```bash
git push origin main
```

### B. Fetch

Downloads latest changes without merging.

```bash
git fetch origin
```

### C. Pull

Fetches and merges changes.

```bash
git pull origin main
```

### B. After Fetching

## Best Practices

### A. Commit Frequently

Small commits are easier to manage.

### B. Write Clear Commit Messages

Bad:

```text
fixed stuff
```

Good:

```text
Fixed authentication bug in login module
```

### C. Use Feature Branches

Avoid direct work on the main branch.

### D. Pull Before Push

Reduces merge conflicts.

### E. Never Push Secrets

Do not push:

- API keys
- Passwords
- Environment files
