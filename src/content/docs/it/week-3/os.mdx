## Memory Management

In a uni-programming system, main memory is divided into two parts: one for the operating system (resident monitor, kernel) and one for the user program currently being executed. In a multiprogramming system, the "user" part of memory must be further subdivided to accommodate multiple processes. The task of subdivision is carried out dynamically by the operating system and is known as **memory management**.

---

### A. Binding of Instructions and Data to Memory

Address binding of instructions and data to memory addresses can happen at three different stages:

#### 1. Compile Time
- If memory location is known a priori, the compiler generates **absolute code**.

#### 2. Load Time
- Must generate **relocatable code** if memory location is not known at compile time.

#### 3. Execution Time
- Binding delayed until run time if the process can be moved during its execution from one memory segment to another.
- Requires hardware support for address maps (e.g., base and limit registers).

---

### B. Logical vs Physical Address Space

- An address generated by the CPU → **logical address** (or virtual address)
- An address seen by the main memory unit → **physical address**
- The set of all logical addresses generated by a program → **logical-address space**
- The set of all corresponding physical addresses → **physical-address space**

> Logical and physical addresses are the **same** in compile-time and load-time binding schemes; they **differ** in execution-time binding.

The **Memory Management Unit (MMU)** is a hardware device that maps virtual to physical addresses. In the MMU scheme, the value in the relocation register is added to every address generated by a user process at the time it is sent to memory:

$$
\text{Physical Address} = \text{Logical Address} + \text{Relocation Register Value}
$$

### C. Dynamic Loading

- A routine is **not loaded until it is called**.
- Unused routines are never loaded → better memory-space utilization.
- Useful when large amounts of code handle infrequently occurring cases (e.g., error routines).
- Does **not** require special OS support.

---

### D. Dynamic Linking

- Linking is postponed until execution time.
- A small piece of code called a **stub** is used to locate the appropriate memory-resident library routine.
- The stub replaces itself with the address of the routine and executes it.
- Particularly useful for **libraries**.
- Requires OS support to check if a routine is in the process's memory address space.

---

### E. Overlays

- Keep in memory only those instructions and data needed at any given time.
- Used when a process is larger than the memory allocated to it.
- Implemented by the user — no special OS support needed, but programming design is complex.

---

### F. Swapping

- A process can be **swapped temporarily out** of memory to a backing store (large disk), then brought back for continued execution.
- **Roll out, roll in**: A variant used for priority-based scheduling — a higher-priority process causes the lower-priority one to be swapped out, and restored after.
- Major part of swap time is **transfer time**, which is directly proportional to the amount of memory swapped.
- Modified versions found in UNIX, Linux, and Windows.

---

### G. Contiguous Memory Allocation

Memory is divided into two partitions: one for the OS and one for user processes.

#### 1. Memory Mapping and Protection

- **Relocation register**: contains the value of the smallest physical address.
- **Limit register**: contains the range of logical addresses.
- Each logical address must be less than the limit register; the MMU maps it dynamically by adding the relocation register value.

#### 2. Memory Allocation

- Simplest method: divide memory into **fixed partitions**, each containing exactly one process.
- The OS maintains a table of free and occupied memory.
- Initially, all memory is one large free block — a **hole**.
- When a process requests memory, the OS searches for a hole large enough, splits it if necessary.

**Dynamic storage-allocation strategies:**

| Strategy | Description |
|---|---|
| **First Fit** | Allocates the first hole that is large enough |
| **Best Fit** | Allocates the smallest hole that is big enough |
| **Worst Fit** | Allocates the largest available hole |

- First Fit and Best Fit are most commonly used; all three suffer from fragmentation.

#### 3. Fragmentation

- **External Fragmentation**: Total memory space is sufficient but not contiguous. Reduced by **compaction** (shuffling memory contents to consolidate free space) — only possible when relocation is dynamic at execution time.
- **Internal Fragmentation**: Allocated memory is slightly larger than requested. Reduced by using partitions of different sizes.

---

### H. Noncontiguous Memory Allocation

Processes can be stored in non-contiguous memory locations. Techniques include:

- Paging
- Segmentation
- Virtual Memory / Demand Paging

---

## Paging

Paging divides **physical memory** (RAM) into fixed-size blocks called **frames**, and **logical memory** into blocks of the same size called **pages**. When a process executes, its pages are loaded into any available frames.

---

### A. Key Concepts

- **Logical Address Space**: Set of all logical addresses generated by a process; divided into pages.
- **Physical Address Space**: Set of all physical addresses in RAM; divided into frames.
- **Page Table**: Data structure mapping logical addresses (page numbers) to physical addresses (frame numbers).
- **Page Size = Frame Size** (always a power of 2, e.g., 4 KB, 8 KB).

**Address breakdown:**

$$
\text{Logical Address} = \text{Page Number } (p) \;|\; \text{Page Offset } (d)
$$

$$
\text{Physical Address} = \text{Frame Number } (f) \;|\; \text{Frame Offset } (d)
$$

$$
\text{Physical Address} = (f \ll \text{offset bits}) + d
$$

**Example:**

Given:
- Physical Address = 12 bits → Physical Address Space = $2^{12}$ = 4K words
- Logical Address = 13 bits → Logical Address Space = $2^{13}$ = 8K words
- Page size = Frame size = 1K words

$$
\text{Number of Frames} = \frac{4K}{1K} = 4 = 2^2
$$

$$
\text{Number of Pages} = \frac{8K}{1K} = 8 = 2^3
$$

---

### B. Steps in Paging

#### 1. Divide Memory
- Logical → Pages; Physical → Frames

#### 2. Allocate Pages
- Load pages into available frames.

#### 3. Page Table
- Map logical pages to physical frames.

#### 4. Translate Address
- Convert logical to physical address via MMU.

#### 5. Handle Page Fault
- Load missing pages from disk if not in memory.

#### 6. Run Program
- CPU uses page table during execution.

---

### C. Structure of the Page Table

#### 1. Hierarchical Page Table

Used when the number of pages is very large. A **two-level page table** pages the page table itself.

A logical address is structured as:

$$
\langle p_1,\; p_2,\; d \rangle
$$

- $p_1$: index into the **outer page table**
- $p_2$: displacement within the page of the outer page table
- $d$: page offset

#### 2. Hashed Page Tables

- Applicable for address spaces larger than 32 bits.
- Virtual page number is hashed into a page table containing chains of elements.
- Virtual page numbers are compared in the chain; on a match, the corresponding physical frame is extracted.

#### 3. Inverted Page Table

- One entry per **real page of memory** (not per virtual page).
- Each entry stores the virtual address stored in that real memory location + the owning process ID.
- Decreases memory needed to store the page table, but increases search time on page reference.

---

### D. Hardware Implementation — TLB

The **Translation Look-aside Buffer (TLB)** is a small, fast associative hardware cache used when the page table is large.

- Each TLB entry: a **tag** (virtual page number) and a **value** (physical frame number).
- On access, the item is compared with all tags simultaneously; if found (TLB hit), the frame is returned immediately.

**Effective Access Time (EAT):**

Let $m$ = main memory access time, $\epsilon$ = TLB access time, $\alpha$ = hit ratio:

$$
\text{EAT} = \alpha \cdot (\epsilon + m) + (1 - \alpha) \cdot (\epsilon + 2m)
$$

Without TLB (page table in main memory):

$$
\text{EAT} = 2m \quad \text{(one access for page table + one for the actual data)}
$$

## Segmentation

Segmentation divides a program's memory into **variable-size segments** based on the logical divisions of the program (e.g., code, stack, data, modules).

---

### A. Key Features

- **Variable-sized segments**: Each segment can have a different size representing meaningful logical units.
- **Logical Addressing**: A logical address is a tuple $\langle \text{segment number},\; \text{offset} \rangle$.
- **Segment Table**: Maps segment numbers to base address and limit (size) in physical memory.
- **Protection & Sharing**: Different segments can have distinct access rights (read, write, execute) and be shared.
- **Fragmentation**: Suffers from **external fragmentation** (variable sizes), but **not** internal fragmentation.

---

### B. Address Translation

$$
\text{Physical Address} = \text{Base Address of Segment} + \text{Offset}
$$

If offset $\geq$ limit → **segmentation fault**.

**Example:**

Segment 0 has base = 500, limit = 600.

$$
\text{Physical Address for logical } (0,\; 600) = 500 + 600 = 1100
$$

---

### C. Paging vs Segmentation

| Feature | Paging | Segmentation |
|---|---|---|
| Division | Fixed-size pages | Variable-size segments |
| Addressing | One-dimensional $(p, d)$ | Two-dimensional $(\text{seg}, d)$ |
| Fragmentation | Internal | External |
| Logical Structure | No relation | Matches program structure |
| Protection/Sharing | Less capable | Better |

## Virtual Memory

Virtual memory creates an illusion of a large, continuous memory space even when physical RAM is limited. Part of the **hard disk** acts as extended memory. The OS stores virtual-to-physical address mappings in a **page table**.

---

### A. Demand Paging

Pages of a process are loaded into memory **only when needed** (on demand), rather than all at once.

#### 1. Page Table and Valid/Invalid Bit

- **Valid bit**: page is currently in physical memory (RAM).
- **Invalid bit**: page is on disk. Accessing it causes a **page fault**.

#### 2. Page Fault Handling

Steps when a page fault occurs:

- **Locate** the required page on disk.
- **Find a free frame** in physical memory (or use page replacement if none available).
- **Load** the page from disk into the free frame.
- **Update** the page table — set frame number and valid bit to valid.
- **Restart** the faulting instruction.

#### 3. Page Replacement

When no free frame is available, the OS selects a **victim frame** using a page replacement algorithm:

- **LRU (Least Recently Used)**: Replace the page not used for the longest time.
- **FIFO (First-In-First-Out)**: Replace the oldest page in memory.
- **Optimal**: Replace the page not used for the longest time in the future (theoretical only).

#### 4. Thrashing

Thrashing occurs when a system spends **more time handling page faults than executing processes**, typically because RAM is too small for the active working sets.

$$
\text{Thrashing} \Leftrightarrow \sum_{i} |W_i(t, \Delta)| > \text{Available Frames}
$$

Where $W_i(t, \Delta)$ is the working set of process $i$ over time window $\Delta$.

---

### B. Copy-On-Write (COW)

- When a process is created (e.g., via `fork()`), the OS **shares** parent memory pages with the child, marking them **read-only**.
- On a write attempt, a **page fault** triggers the OS to create a private copy for the modifying process.
- The original page remains shared with the other process.

Steps:
- Parent and child share read-only memory pages.
- A write attempt triggers a page fault.
- OS allocates a new physical page and copies original content.
- New page gets read-write permissions; original remains shared.

---

## Page Replacement Algorithms

When a page fault occurs and no free frame is available:

1. Find the desired page on disk.
2. If no free frame → select a **victim frame** using a replacement algorithm; write it to disk if modified.
3. Load the desired page into the (now free) frame.
4. Update the page and frame tables.
5. Restart the user process.

Performance is evaluated by counting **page faults** on a given **reference string**.

---

### A. FIFO (First-In-First-Out)

- Replaces the **oldest** page in memory.
- Pages are managed in a queue; the head is replaced, new pages join the tail.
- Suffers from **Bélády's anomaly**: page fault rate can *increase* as more frames are allocated.

### B. Optimal Page Replacement (OPT / MIN)

- Replaces the page that **will not be used for the longest period** in the future.

- Does not suffer from Bélády's anomaly.

### C. LRU (Least Recently Used)

- Replaces the page **not used for the longest time** (past usage as proxy for future).
- Uses **timestamps** — on page access, timestamp is updated; on replacement, the page with the oldest timestamp is evicted.
- More accurate than FIFO, but **expensive to implement** (requires stacks or linked lists to track order).

### D. LRU Approximation

Uses **reference bits** in the page table (set to 1 by hardware on access, cleared periodically by OS).

#### 1. Additional-Reference-Bits Algorithm

- Each page keeps an **8-bit shift register**.
- At regular intervals, the OS shifts the reference bit into the high-order bit; remaining bits shift right.
- Page with the **smallest unsigned integer value** is considered the LRU page.

#### 2. Second-Chance Algorithm

- Based on FIFO; before replacing a page:
  - If reference bit = **0** → replace it.
  - If reference bit = **1** → clear the bit, give it a second chance, move to next FIFO page.
- A page given a second chance will not be replaced until all others are considered first.

## Frame Allocation

### A. Equal Allocation

Divide $m$ frames evenly among $n$ processes:

$$
\text{Frames per process} = \left\lfloor \frac{m}{n} \right\rfloor
$$

### B. Proportional Allocation

Allocate frames in proportion to process size. Let $s_i$ = virtual memory size of process $p_i$, $S = \sum s_i$, and $m$ = total frames:

$$
a_i = \frac{s_i}{S} \times m
$$

---

### C. Global vs Local Replacement

- **Global replacement**: A process can take a frame from any other process.
- **Local replacement**: Each process selects only from its own allocated frames.

---

## Thrashing

Thrashing is a condition where the system spends **more time swapping pages** than executing processes, typically caused by too many processes competing for insufficient RAM.

---

### A. Causes

- High degree of multiprogramming.
- Poorly designed page replacement algorithms (e.g., naive FIFO).
- Physical memory too small to hold the working sets of all active processes.

---

### B. Detection — Working Set Model

For each process, the OS tracks its **working set** $W(t, \Delta)$: the set of pages referenced in the time interval $(t - \Delta,\; t)$.

$$
\text{Thrashing occurs if } \sum_i |W_i(t, \Delta)| > \text{Total Available Frames}
$$

Additional signals:
- **High page-fault rate** (many faults per second).
- **Low CPU utilization** (CPU often idle waiting for page loads).

---

### C. Prevention Strategies

#### 1. Local Page Replacement
- Restrict processes to replacing only their own frames.
- Prevents one large process from consuming all memory and causing thrashing in others.

#### 2. Working Set Model
- Monitor the working set of each process.
- If total working set size exceeds available memory, reduce degree of multiprogramming by swapping out some processes.

#### 3. Page-Fault Frequency (PFF) Scheme
- Monitor each process's page-fault rate and adjust frame allocation accordingly:
  - Page faults **too high** → allocate more frames.
  - Page faults **too low** → reclaim some frames.
