# High Level Design (HLD)
## Introduction to HLD

High Level Design (HLD) focuses on the overall architecture and structure of software systems.

HLD mainly deals with:

- System architecture
- Scalability
- Databases
- APIs
- Traffic handling
- Communication between services

HLD does not focus on:

- Low-level coding
- Internal function logic
- Small implementation details

Examples of systems requiring HLD:

- YouTube
- Instagram
- Netflix
- WhatsApp
- Amazon

## Functional vs Non-Functional Requirements

Requirements are mainly divided into two types.

### A. Functional Requirements

Functional requirements describe:

- What the system should do
- Features provided by the system

#### 1. Example

For Instagram:

- Upload posts
- Like posts
- Follow users
- Comment on posts

### B. Non-Functional Requirements

Non-functional requirements describe:

- Scalability
- Reliability
- Availability
- Security
- Performance

#### 1. Example

For Instagram:

- Feed loading should be fast
- Millions of users should be supported
- System should remain highly available



## Scalability

Scalability is the ability of a system to handle increasing traffic efficiently.

A scalable system can continue performing properly even when users and requests increase.

### A. Vertical Scaling

Vertical scaling means increasing server power by upgrading:

- CPU
- RAM
- Storage

#### 1. Example

8 GB RAM → 64 GB RAM

#### 2. Advantages

- Simple to implement
- Easier management

#### 3. Limitations

- Hardware limits exist
- Expensive

### B. Horizontal Scaling

Horizontal scaling means adding multiple servers.

#### 1. Example

1 server → 10 servers

Horizontal scaling is more commonly used in modern distributed systems.

#### 2. Advantages

- Better scalability
- Better fault tolerance

#### 3. Limitations

- Complex architecture
- Requires load balancing

## Load Balancer

A Load Balancer distributes incoming traffic across multiple servers.

### A. Why Load Balancers are Important

Without load balancing:

- One server may crash due to heavy traffic

With load balancing:

- Traffic is distributed evenly

### B. Benefits

- Prevents server overload
- Improves scalability
- Improves availability
- Reduces downtime

### C. Common Algorithms

#### 1. Round Robin

Requests are distributed one after another sequentially.

#### 2. Least Connections

Traffic is sent to the server with the fewest active connections.

#### 3. IP Hashing

Requests are routed based on client IP address.

#### 4. Example

YouTube distributes traffic across multiple servers using load balancers.

## CAP Theorem

CAP Theorem states that distributed systems can guarantee only two out of these three properties at the same time:

- Consistency
- Availability
- Partition Tolerance

CAP theorem becomes important when network failures happen between servers.

### A. Consistency

Consistency means:

All users see the same latest data.

#### 1. Example

In banking systems:

- All ATMs should show updated balances immediately after transactions.

### B. Availability

Availability means:

The system should always respond to requests.

#### 1. Example

Instagram should still load feeds even during partial server failures.

Some data may not be instantly updated.

### C. Partition Tolerance

Partition Tolerance means:

The system should continue working even if communication between servers fails.

#### 1. Example

If servers in India and the USA lose connection:

- The system should still continue functioning.

### D. CP Systems

CP systems prioritize:

- Consistency
- Partition Tolerance

#### 1. Example

Banking systems

### E. AP Systems

AP systems prioritize:

- Availability
- Partition Tolerance

#### 1. Example

Instagram likes and YouTube views.

### F. Eventual Consistency

Eventual consistency means:

Data may not update instantly everywhere,
but eventually all servers become synchronized.

## System Design Process

A structured process is followed while designing systems.

### A. Understand Requirements

Identify:

- Functional requirements
- Non-functional requirements

### B. Estimate Scale

Estimate:

- Number of users
- Traffic
- Storage requirements

### C. Design Core Components

Identify:

- Database
- Cache
- APIs
- Load balancers

### D. Create Architecture

Design interaction between components.

### E. Identify Bottlenecks

Find possible:

- Performance issues
- Scalability problems
- Failure points



## Solved Example — Design a URL Shortener (Bit.ly)

A URL Shortener converts long URLs into short and manageable links.

### A. Example

Long URL:

https://www.example.com/products/mobile/samsung/offer

Short URL:

bit.ly/abc123

The shortened URL redirects users to the original long URL.

The system should support:

- High traffic
- Fast redirects
- Scalability
- Reliability



### B. Functional Requirements

The system should:

- Generate short URLs
- Redirect users
- Generate unique short links
- Handle millions of redirects
- Support high read traffic



### C. Non-Functional Requirements

The system should provide:

- High availability
- Low latency
- Scalability
- Reliability

#### 1. Why Low Latency is Important

When users click a short URL:

- Redirection should happen instantly

Slow redirects create poor user experience.



### D. Core Components

Main components include:

- Client
- Load Balancer
- API Servers
- Cache
- Database

### E. Client

The client can be:

- Browser
- Mobile application
- Web application

The client sends requests to:

- Create short URLs
- Open short URLs



### F. Load Balancer

The Load Balancer distributes requests across multiple API servers.

#### 1. Benefits

- Prevents server overload
- Improves scalability
- Improves availability

When millions of users click shortened URLs simultaneously:

- Traffic gets distributed evenly across servers.



### G. API Servers

API servers handle:

- URL creation
- URL redirection

#### 1. Responsibilities

- Accept long URLs
- Generate short codes
- Store mappings
- Redirect users

API servers also validate:

- Invalid URLs
- Duplicate requests
- Malicious links



### H. Database

The database stores mappings between:

| Short URL | Original URL |
| --- | --- |
| abc123 | https://example.com |

The database must support:

- Fast reads
- Fast writes
- Scalability

Read traffic is usually much higher than write traffic because:

- Links are clicked more often than created.


### I. Cache

Cache stores frequently accessed URLs temporarily.

#### 1. Why Cache is Important

Popular URLs may receive millions of clicks.

Fetching from the database every time:

- Increases latency
- Increases database load

Using cache:

- Improves speed
- Reduces database traffic

#### 2. Popular Cache Systems

- Redis
- Memcached

#### 3. Example

If a viral YouTube link is accessed millions of times:

- Store the mapping in cache for faster redirects.

### K. URL Redirect Flow

#### 1. Step 1

User opens the short URL.

#### 2. Step 2

System checks cache first.

##### a. If Present in Cache

- Redirect immediately

##### b. If Not Present

- Fetch from database
- Store in cache
- Redirect user

This improves:

- Speed
- Scalability
- Performance


### L. HTTP Redirects (301 vs 302)

When users open a shortened URL,
the server redirects them to the original URL using HTTP redirect status codes.

#### 1. 301 Redirect

301 means:

**Permanent Redirect**

The browser understands that:

- The URL has permanently moved.

##### a. Advantages

- Better SEO
- Browser caching
- Faster future redirects

##### b. Example

bit.ly/abc123 permanently redirects to the original URL.

#### 2. 302 Redirect

302 means:

**Temporary Redirect**

The browser understands that:

- The redirect is temporary.

##### a. Advantages

- Useful for temporary routing
- Useful for testing

##### b. Example

Temporary maintenance pages or testing environments.

#### 3. Difference Between 301 and 302

| 301 Redirect | 302 Redirect |
| --- | --- |
| Permanent | Temporary |
| Better SEO | Not ideal for SEO |
| Browser caches redirect | Usually not cached |

#### 4. Which One is Used in URL Shorteners?

Most URL shorteners use:

- 301 Redirects

because shortened links usually permanently point to the same URL.



### M. How Short Codes are Generated

The system must generate:

- Unique codes
- Small URLs
- Collision-free values

#### 1. Hashing

Generate hash values from URLs.

##### a. Problem

Hash collisions may occur:

- Two URLs may generate similar values.

#### 2. Base62 Encoding

Uses:

- a-z
- A-Z
- 0-9

This creates compact and readable URLs.

##### a. Example

125 → cb

#### 3. Why Base62 is Used

Base62 allows:

- Small URLs
- Large combinations
- Easy readability
